  1. Introduction

1.1 Purpose of this Document

This document describes the requirements for the Sync4j SyncML server and framework. It represents the basis for the development of the Sync4j project and the description of the features and functionality end users, IT managers and developers can expect from Sync4j.

1.2 Related Documents

The following documents are related to this work:

  • SyncML protocol specifications

  • Sync4j DM Requirements document (available soon)

  • Sync4j Architecture document

  • Sync4j Implemented Requirements

  • Sync4j Coding Standards

Overview of Sync4j

Sync4j is a java implementation of a Synchronization Markup Language (SyncML [1]) server and framework.


SyncML is defined on the SyncML website as:


SyncML is the open standard that drives data mobility by establishing a common language for communications between devices, applications and networks. The foundation of the SyncML open standard, SyncML Data Sync (SyncML DS) ensures a consistent set of data that is always available on any device or application, any time. SyncML Device Management (SyncML DM) enables OTA administration of devices and applications, simplifying configuration, updates and support.


For further information on SyncML in general, please refer to http://www.syncml.org/.


Sync4j provides a framework to create, send, receive and process the messages that make up a SyncML session. Together with a simple example, the framework can directly be used for synchronization. By providing classes that implement the Sync4j interfaces, you make Sync4j use any data source and type, solve conflicts between local and remote data as you like etc.


Sync4j is at the same time a complete product (the SyncML server) and a framework that developers can extend or reuse in order to implement their own server and synchronization service.

Sync4j aims to implement the two main SyncML protocol families: SyncML Data Sync (SyncML DS) and SyncML Device Management (SyncML DM).

1.3 Business Context

Sync4j is an open source project developed under the BSD license (see http://www.opensource.org/licenses/bsd-license.php). This means you can use the code in any application you like, modify and redistribute it, as long as you leave the copyright notice and warranty disclaimer.


As an open source project, there is no guarantee for anything to work as intended. This said, we will work hard to make Sync4j as stable as possible and consider any bug reports, security considerations etc. submitted by users.


Sync4j is hosted at sourceforge.net (http://sync4j.sourceforge.net).

  1. General Description

This section describes in detail what Sync4j is.

2.1 Product Features

Sync4j is at the same time a SyncML server implementation and a framework that users and developers can use to build their own synchronization services. It will implement both the data synchronization and device management parts described in the SyncML standard.


Sync4j currently supports only HTTP as transport protocol. Other protocols may be added in the future if the Sync4j user community will require it.


Being 100% java, Sync4j can be used on any platform with java support as soon as a JVM ver. 1.4 or later is available.

2.1.1 Client

Developing a complete SyncML client supported on the most different devices is out of the scope of Sync4j. As a standard developed, proposed and adopted by a consortium of the most important mobile manufacturer companies, SyncML will be directly implemented in the majority of mobile devices such as mobile/smart phones, PDA, etc. Therefore, trying to create a generic client working from devices with a very constrained footprint (few Kilobytes) to powerful pocket and tablet pc, would be an effort wasted as soon as the device will support directly SyncML. Due to the limited available resources, Sync4j focuses on the server-side aspects. Nevertheless, a basic thick-client SyncML will be implemented as part of the project for the sake of testing and integration with other systems.

However the support of different Java platform and capability might be added in the future, based on the requests of the Sync4j community (See also section General Constraints). If you would like to implement such a thin client with Sync4j, please contact the Sync4j team.

2.1.2 Server

Sync4j will implement a SyncML DS/DM full featured server. It is designed to run inside any Java 2 Enterprise Edition (J2EE) application server such as the open source JBoss or commercial Bea Weblogic, SunOne, etc.

The transport protocol is implemented as a servlet used to get the HTTP POST message and pass it into the protocol framework, which in turn, runs inside an Enterprise Java Bean. This architecture takes advantage of the Quality of Service (QoS) features provided by the hosting application server in order to guarantee high scalability and reliability.

2.1.3 Framework

The Sync4j server is built on top of a framework developed with modularity and extensibility in mind. The goal is that each module should be replaceable, pluggable and extensible. The most important modules are:

  • client

  • core

  • engine

  • protocol

  • server

These modules are described in more detail in the following sections.

2.2 Similar System Information

Sync4j is based on Java technologies and platforms.

The framework in itself can be used in any J2SE version 1.4 or later environment as a standalone program.


The server, instead, is designed to be deployed in a J2EE version 1.3 or later application server. This is for the following reasons:

  • It avoids to care about server-side collateral services such as multithreading, tcp/ip connections, data marshalling and unmarshalling, transactions, session management, etc.

  • The J2EE platform has gained a wide acceptance in the enterprise IT infrastructure, thus Sync4j can be plugged in existing environments with a minimal impact.

  • It does not require the development of a standalone server that IT managers are forced to install, deploy and maintain.

User Characteristics

From the users standpoint, Sync4j can be viewed as a complete synchronization solution whenever it meets their requirements of data synchronization. If features are missing or need customization, programmers can use the framework to extend or build concrete applications on top of it. Developers can eventually decide to use only the framework, developing from scratch an entire SyncML server.

Programmers need no detailed knowledge about SyncML. It can be used for any application needing to synchronize data of a record-like structure with a server and for any server that needs to offer synchronization functionality.

2.3 User Problem Statement

Many applications need to synchronize data records. Some servers or clients operate on standard formats like contact information (vCard), calendars (vCalendar) or emails; others on custom formats for time reports, stock updates or whatever.

Developing a new synchronization protocol for every application is not desirable. While for applications with proprietary servers and clients, this could be an option, for standard services this is not possible at all. This is where SyncML comes into play. Any device with SyncML and knowledge of the data type can synchronize with any server supporting SyncML.


Sync4j is designed to run on any java enabled device. It allows to implement SyncML clients and servers with little knowledge about the caveats of synchronization. Further, programs will be interoperable with all other SyncML services.

2.4 User Objectives

Sync4j is easy to use. Developers can look at the implementation and get a running application for their purposes very quickly. For SyncML servers, all that needs to be written is data access (persistent storage). For the client, the data access mostly is not enough, as probably the application will also offer a user interface. How the data is generated (user inputs, sensor read etc.) is out of the scope of Sync4j.


Generally speaking the goals that Sync4j wants to achieve are:

  • Provide a full featured and ready to use SyncML DS/DM server.

  • Support the major data format used by today's mobile applications (i.e. vCard, vCal and other PIM data).

  • Support for any kind of data business mobile applications may require (by a concept of pluggable data access).

  • Provide an easy to use synchronization system for the final user.

  • Provide easy integration with existing systems, especially in the telecommunications field.

  • Provide a flexible, pluggable and easy to use framework for the development of synchronization services.

General Constraints

One of the aspects debated in the Sync4j community was how much to rank the support of java SyncML clients for limited devices based on the Java 2 Micro Edition technology. Some work was also done in order to make the core classes J2ME compliant. Some people supported the vision that Sync4j should provide a client implementation working on existing J2ME mobile devices, whilst others had the opinion that the Sync4j client framework should address thick java clients.

At the present time, the choice has fallen on the latter option. In fact, even if J2ME clients must be kept in consideration for future releases, the constraints imposed by the todays implementation of J2ME (essentially based on the Mobile Information Device Profile - MIDP) do not permit the development of a reasonable SyncML client framework. J2ME/MIDP is targeted to tiny devices, with a very limited memory and resource footprint, which never could meet the requirements of an even limited SyncML client. To give an idea of the limitations the MIDP profile imposes to any compliant program, it is sufficient to remark that a midlet suite should not be bigger than 30Kb.


However, the discussion is still open, since J2ME is evolving from supporting very thin and light devices (based on the MIDP profile), to more complete and powerful platforms based on the Foundation and Personal profiles. In addition, many more powerful devices will implement the SyncML client in the firmware or in the operating system, exposing a proper API to custom applications. For example, Symbian 7.0 [2] includes a SyncML client agent and API.

The issue is even more true in the case of SyncML DM. Device management will be tightly integrated in the device firmware/operating system and manufactures will implement a SyncML DM client agent customized to expose each device's specific features.

Thus, given the limited development resources, the priority for Sync4j is currently to provide a fully featured SyncML server and a SyncML framework for J2SE clients. However, Sync4j is open to contributions of any volunteer and the addition of a version of the client agent framework targeted to limited devices is welcome.

3.2.1 Accessing External Data Sources

The requirement EG0002 needs a bit more specification. It is a main intent of Sync4j to be a very flexible and modular server, easy to integrate in and with existing systems. One key aspect of this strategy is to be able to access data to synchronize wherever they are and in whatever form they are.

Sync4j should be able to work with the most disparate synchronization sources (SyncSources), in a completely transparent way, provided an appropriate connector to the data. This connector should provide at least the following operations:

  • Retrieve all items for a given user

  • Retrieve the new items for a given user since the last synchronization

  • Retrieve the updated items for a given user since the last synchronization

  • Retrieve the deleted items for a given user since the last synchronization

  • Add new items for a given user

  • Update items for a given user

  • Delete items for a given user

Description

Access to persistent storage must be encapsulated with a simple interface. Having specified the data access classes and authentication classes (see point 5), Sync4j should be able to run with no further programming. Standard cases like JDBC and filesystem are implemented as example and as quick start.

Critically

Developers don't want to dig deep into Sync4j and SyncML protocol specifications. The easier this feature is, the better Sync4j is usable. Not needing to fuss with Sync4j more deep also helps getting stable sycn4j applications as a programmer quickly hacking a working app together could make errors if he would need in-deep knowledge of SyncML

Risks:

To get this work, the interface must be at the same time simple but also complete. For properly configuring the framework and easy instantiation of the access classes, design patterns (e.g. factory) could be used.

3.3 Sync4j Server

The Sync4j server is a J2EE application. Developers should not need to re-implement the server as the only thing that varies is the (configurable) data access. However, the framework will allow it if someone wants to do it anyway.

As a result of basing Sync4j on the J2EE architecture, the server will inherit the "quality of service" features provided by the application server. Moreover, IT managers and administrators will not have to install and manage a separate server in a different infrastructure; they can instead reuse the existing infrastructure. Plus Sync4j may take advantage of the AS's manageability functions such as SNMP and JMX support.

Sync4j will fully support HTTP as transport protocol. It is the widest adopted protocol and it is available from mobile phones to enterprise servers. Other transport protocols may be taken into consideration in the future if the Sync4j community will require it.


Both XML and WBXML message formats will be supported.

The Syn4j server will be able to manage users and roles as described in a following section.

From a configuration point of view, Sync4j uses as final configuration repository a set of XML and properties files. It should be possible to change the configuration of the server without interrupting the service.


Description

The implementation of the Sync4j server

Critically

Technical issues:

The server must be easy to configure and reconfiguration should happen as much as possible without restarting.

  1. User Requirements

In the first version of Sync4j, users are divided in three main categories: Administrator, Manager and Client. These are represented by the roles "Administrator", "Manager", "Client". Sync4j will support many storage media for users through the features FK0012 and SR0006. The goal is that the server, after a simple reconfiguration, should be able to work with users stored in the database, in a directory server, in a file located on the file system, etc.

The following information is collected for every user:

  • Unique id

  • First name

  • Last name

  • Email

Additional information depends on the user role.


The following users are created at the installation of Sync4j:

4.1 User Roles

Three user roles are identified at this stage: "Administrator", "Manager" and "Client".

4.1.1 Administrator Role

Users with role Administrator have the right of administering the Sync4j server. They will have access to all the server configuration, administration and monitoring interfaces and tools. They will not have the rights to see and manage other users, but they can create new managers.

4.1.2 Manager Role

Users with role Manager are able to create new users and to see and update user characteristics. They can also delete users.

4.1.3 Client Role

Users with role Client are the final users. They are able to access the synchronization services user managers allow them. For each user the following information is additionally recorded:

  • Devices in use (a list of device id)

Interface Requirements

4.2 User Interfaces

The first release of Sync4j will not provide a configuration user interface. The configuration of the server will be done via properties and configuration files.

4.2.1 Command Line Interface

None.

4.2.2 Diagnostics

The Sync4j server will provide the capability to log various levels of system detail for integration, performance tuning and debugging purposes. Statistical data will be provided to support performance tuning. Logging is based on the logging subsystem of the J2SE 1.4 platform.

4.3 Hardware Interfaces

Sync4j is 100% pure java. It must not depend on the hardware, only on a working java implementation for this hardware.

4.4 Communications Interfaces

The device running a Sycn4j client or server must be able to offer some sort of communication port. This is encapsulated in a generic connection framework, thus independent of the connection. Typical case is a HTTP connection.

4.5 Software Interfaces

Sync4j and especially the Sync4j framework define numerous interfaces:

  • Interface for persistent storage access (id mapping, sync anchor, server specific data)

  • Interface for the synchronization engine

  • Interface for sync sources

  • Interface for authentication/authorization

4.6 Security

User authentication in Sync4j is encapsulated with a simple interface. It allows to use any authentication back-end and any desired system like md5 or basic (see SyncML specifications).

The following authentication adapters are provided:

  • no authentication

  • db authentication

  • JAAS authentication


Description

Sync4j security services

Critically

The preferred way to approach authentication/authorization in Sync4j is by the mean of the JAAS adapter. This represents the standard way a java application accesses security services. It is already a pluggable and configurable architecture that seamlessly integrate in Sync4j.

4.8 Maintainability

Since one of the main design goals of Sync4j is modularity and flexibility, it is foreseeable that will require lot of configuration files. In order to keep the configuration of Sync4j manageable, a configuration architecture will be provided. It will furnish a standard and homogenous way of reading configuration files.

4.9 Portability

Portability is guaranteed by the adoption of the J2SE v1.4 (or later) and J2EE v1.3 (or later) platforms.

4.10 Extensibility

One of the most important goal of Sync4j is to provide a modular and extensible framework that users can use out of the box or easily adapt to their needs. It must be very simple to extend it to synchronize any type of data and sources and to integrate in the existing Information Technology infrastructure.

4.11 Packaging and Installation

A quick and easy to use installation procedure must be provided. A GUI is not strictly required; a command line installation is sufficient as soon it is composed of very few and necessary steps. The installation package must include everything required to start the Sync4j server. This includes:


- Database

- Application Server

- Required libraries

- JRE


A guide for the installation and deployment on the supported application servers and databases will be provided.
